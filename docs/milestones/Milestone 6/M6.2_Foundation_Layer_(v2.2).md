# M6.2 — Foundation Layer (v2.2)

We are updating M6.2 for mediCalm.

This update must remain fully consistent with M6.0 and M6.1.

## Objective

Define the routing, schema, typing, and context foundation required for M6.

---

## 1. Entry State Type

```ts
export type EntryState =
  | 'pain'
  | 'anxious'
  | 'angry'
  | 'sad'
  | 'exhausted'
  | 'tight'
  | 'overwhelmed'

Requirements:

state_entry must be typed as EntryState[]
Do not leave this implied
Do not allow alternative string values
2. Session Type Union

Extend the existing session_type union explicitly.

type SessionType = 'HARI' | 'LEGACY' | 'STATE'

Requirements:

Reuse the existing session_type field
Do not create a parallel session-type field
Preserve backward compatibility
3. M6 Schema Additions

This section defines M6 additions only, not the full STATE session contract.

// M6 additions — backward-compatible

session_type: 'HARI' | 'LEGACY' | 'STATE'

state_entry?: EntryState[]           // present when session_type === 'STATE'
outcome_primary?: 'better' | 'same' | 'worse'
outcome_marker?: string              // per-state values defined later (M6.6)

Requirements:

outcome_marker remains a string for now
Do not prematurely create a fixed enum
outcome_primary must be explicitly defined as 'better' | 'same' | 'worse'
If an existing SessionResult type exists, either reuse Exclude<SessionResult, 'interrupted'> or define outcome_primary separately with a note that interruption is handled elsewhere
4. Pain Field Handling

Pain-centric fields must be explicitly null, empty, or omitted on session_type: 'STATE' entries.

Finalized handling
location_tags → []
symptom_tags → []
pain_level → null
flare_sensitivity → null
symptom_focus → null

Requirements:

location_tags and symptom_tags remain required arrays to preserve type safety
Do not write placeholder values into pain-trained fields
All non-pain shared fields (timestamps, ids, session metadata, etc.) remain valid
5. hari_metadata Handling

STATE sessions must omit hari_metadata entirely.

Requirements:

Do not populate hari_metadata with null-equivalent values
Do not partially reuse pain-centric HARI metadata structures for STATE sessions
6. AppContext Additions

Add the minimum new AppContext state needed for the M6 routing chain.

pendingStateEntry: EntryState[] | null

Purpose:
This preserves selected states across:

HomeScreen
→ StateSelectionScreen
→ SADSafetyScreen
→ Intake

Lifecycle:

set on Continue from StateSelectionScreen
preserved through the SAD safety flow
cleared on session save
cleared on Return Home from escalation exit
cleared on full home reset

Requirements:

Do not leave this implicit
Do not store intensity in pendingStateEntry
7. Navigation Screen Identifiers

These are additions to the existing AppScreen union / screen state machine, not a replacement.

type AppScreen =
  | 'StateSelection'
  | 'SADSafety'
  | 'SupportResources'

Requirements:

Preserve all existing screen identifiers already used by the app
Add these new M6 identifiers consistently to the existing navigation model
Do not redefine AppScreen as only these three values unless the codebase already composes screen types differently
8. Routing

Minimum M6 path:

Home
→ StateSelection
→ SADSafety (conditional)
→ Intake

Requirements:

Keep existing navigation stable
Do not break legacy flows unless explicitly required
M6 path begins from state selection
9. Intensity Handling

Clarification:

Intensity is not stored in pendingStateEntry
Intensity is collected later in M6.3 (Dynamic Intake Screen)
Intensity is passed separately to HARI after intake
10. User Profile Name
name?: string

Requirements:

Optional-safe only
No hardcoded name assumptions

Capture:

Name capture is not implemented in this milestone

Fallback behavior:

If no name is available, render the header without forced placeholder personalization

Example fallback:

What are you feeling right now?

Do not invent greetings like “Hi there” unless already consistent with the product.

Constraints

Must:

preserve backward compatibility
separate STATE from pain logic
keep routing deterministic
make M6 data structures ready for later intake, completion, and pattern-memory work

Must not:

implement intake logic
implement HARI routing logic
implement completion logic
invent extra fields not required by M6 architecture
collapse STATE sessions into pain-session assumptions
Identity Lock

This is infrastructure only.

No emotional interpretation layer is introduced.

mediCalm remains a physiological regulation system with state-aware entry.
