Treat M6.8.3.1 as a conservative refinement layer, not a personalization engine.

Do NOT re-implement M6.

This is a targeted implementation milestone.

Goal:
Use recent HARI feedback to make small, controlled adjustments to the next session recommendation without making the system unpredictable.

----------------------------------------
ARCHITECTURE POSITION

Current pipeline:
input -> NeedProfile -> FeasibilityProfile -> BreathFamily -> BreathPrescription -> runtime

M6.8.3.1 adds:
recent HARI feedback -> controlled adjustment -> final BreathPrescription

This is NOT a new architecture.
It is a thin adaptation layer on top of the existing pipeline.

----------------------------------------
M6.8.3.1 RULES

1. Change only when signal is clear
2. Change only one major variable at a time
3. Never override current feasibility
4. Only adapt strongly when the new state is similar to the recent prior state
5. Prefer stability over novelty

----------------------------------------
FEEDBACK SIGNALS

Use the subtle shift model already established:

- relaxed
- open
- steady
- energized
- no_change
- tense_tight

These are state-shift signals, not scores.

----------------------------------------
ADAPTATION LOGIC

Positive signals:
- relaxed -> keep same family
- open -> keep decompression-oriented family
- steady -> keep stabilization-oriented family
- energized -> keep activation family if current state is similar

Neutral:
- no_change -> do NOT switch family immediately
  First adjust one delivery variable only:
  - duration OR
  - pacing OR
  - session softness
  Do not change more than one major variable.

Negative:
- tense_tight -> simplify immediately
  First response:
  - reduce intensity/load
  - shorten duration
  - remove structure if needed
  Only switch family if needed after simplification logic.

----------------------------------------
SIMILARITY RULE

Adapt strongly only if current session is similar to the prior session.

Similarity should be based on:
- primary need
- primary state
- sensitivity band / safety level
- mixed vs single-state condition

If not similar:
- do not apply strong adaptation
- default mostly to current-session logic

----------------------------------------
ONE-VARIABLE RULE

M7.1 may adjust only ONE major variable per session:

Allowed major variables:
- family
- duration
- pacing / softness

If a session was positive:
- keep family
- optional slight duration increase only if safe

If neutral:
- keep family first
- adjust duration or pacing only

If negative:
- simplify first
- only then consider family switch

Do NOT change:
- family + duration + pacing all at once

----------------------------------------
ALLOWED FAMILY SWITCHING

Use stabilization as the safest bridge family.

Safe examples:
- activation -> stabilization
- decompression -> stabilization
- downregulate -> stabilization
- stabilization -> decompression
- stabilization -> downregulate
- stabilization -> activation

Avoid aggressive direct jumps unless clearly justified by current state.
