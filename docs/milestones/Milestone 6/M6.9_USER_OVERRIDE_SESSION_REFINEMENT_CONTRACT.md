# M6.9_USER_OVERRIDE_SESSION_REFINEMENT_CONTRACT.md

## Status

FINAL — Locked for implementation after M7.3

This contract defines a **bounded user refinement layer** that operates strictly:
- after M6 prescription
- before M7 delivery

All corrections from architecture review have been applied.

---

## 1. Purpose

M6 produces a valid, safe, deterministic session based on:
- state selection
- intake
- feasibility
- interpretation
- controlled adaptation

M6.9 exists to reduce mismatch between:

- system-derived prescription
- user's present-moment felt need

This layer allows:
> small, safe, controlled refinement

---

## 2. Placement in Architecture

```text
M6 Input + Intake
  -> M6 Interpretation
  -> M6 Prescription / Session Config
  -> M6.9 User Refinement
  -> M7 Guidance Delivery
  -> Runtime
```

Non-negotiable:

- M6.9 never moves earlier
- M6.9 never bypasses M6
- M6.9 never replaces interpretation

---

## 3. Authority Boundaries

**M6 owns:**
- interpretation
- feasibility (including environment constraints)
- breath family
- safety rules
- prescription validity

**M6.9 owns:**
- bounded refinement of delivery-relevant parameters

**M7 owns:**
- cue timing
- cue wording
- silence density
- delivery behavior

---

## 4. Core Principle

M6.9 is:

**refinement, not control**

It must feel like:

- nudge
- soften
- align

Never:

- configure
- build
- program

---

## 5. Non-Negotiable Constraints

M6.9 must never:

- override safety
- override feasibility
- introduce invalid breathing patterns
- expose engine complexity
- duplicate intake
- allow custom breath programming
- introduce counts
- collapse M7 responsibilities

If conflict occurs:

**M6 always wins**

---

## 6. Entry Trigger

M6.9 is:

- optional
- post-prescription only
- not required
- not part of intake

UI Placement:
```
[ Begin Session ]
[ Refine ]
```

Rules:

- secondary action only
- never blocks start
- no auto-entry

---

## 7. Input Model (Locked)

**Format**

Single-screen refinement surface

**Interaction Rules:**
- 0–2 selections maximum
- no required inputs
- no step flow
- no typing
- no scrolling forms

---

## 8. Allowed Inputs

### A. Felt Need Bias

Prompt:
> What do you need more of?

Options:

- Expand
- Calm
- Activate (conditionally shown)
- Decompress
- Keep it gentle

### B. Delivery Intensity

Prompt:
> How should this feel?

Options:

- Very gentle
- Standard
- Slightly stronger

### C. Start Style

Prompt:
> How should this begin?

Options:

- Ease in
- Standard start

---

## 9. Removed Inputs

The following are NOT part of M6.9:

- Environment (moved to M6.7 feasibility)
- Pain maps
- Sensitivity sliders
- Custom breathing inputs
- Any intake duplication

---

## 10. Activation Visibility Rule

If activation is disallowed by M6:

- "Activate" must not be shown

Optional fallback:

- "Keeping this session gentle"

Default behavior:

- hide option entirely

---

## 11. Adjustment Scope

M6.9 may adjust:

- entry softness
- delivery softness
- cue density bias
- body emphasis
- intensity (within bounds)
- support suggestion flag

---

## 12. Disallowed Adjustments

M6.9 must not modify:

- state interpretation
- need hierarchy
- feasibility logic
- breath family
- safety constraints
- learning records
- structural session format

---

## 13. Refinement Model

```text
baseSessionConfig
+ refinementInputs
+ guardrails
= refinedSessionConfig
```

Rules:

- deterministic
- repeatable
- bounded

---

## 14. Guardrails

All refinements pass:

**Safety Check**

→ block invalid changes

**Feasibility Check**

→ clamp to allowed range

**Architectural Check**

→ prevent scope violations

---

## 15. Output Contract (Typed)

```typescript
type SessionRefinement = {
  entryStyleBias?: "gentle" | "standard"
  deliverySoftness?: "very_gentle" | "standard" | "slightly_stronger"
  cueDensityBias?: "reduced" | "standard"
  bodyEmphasis?: "expand" | "calm" | "decompress" | "activate"
  intensityAdjustment?: -1 | 0 | 1
  supportSuggestion?: boolean
}
```

Rules:

- deterministic
- bounded
- M7-compatible
- no expansion over time

---

## 16. Support Suggestion Definition

A lightweight suggestion such as:

> "Rest your arms if that feels easier"

Must not:

- instruct posture correction
- force repositioning
- interrupt session flow

---

## 17. Memory-Informed Sessions

If session originates from "Continue What Helped":

- Refinement must be low influence only

**Clamp Rules:**
- intensityAdjustment ∈ [-1, +1]
- no breath family changes
- no major delivery shifts
- preserve base session identity

---

## 18. Relationship to M7

M6.9 may influence:

- cue density
- entry softness
- emphasis framing

M7 controls:

- timing
- wording
- silence
- delivery execution

---

## 19. Relationship to M8

Refinement ≠ truth

M6.9 must NOT:

- update learning directly
- treat preferences as effectiveness

M8 may observe patterns over time.

---

## 20. Relationship to M9

- M9 = fast path
- M6.9 = precision path

M6.9 must remain:

- optional
- lightweight
- non-blocking

---

## 21. Failure Conditions

M6.9 fails if it:

- duplicates intake
- allows manual breath creation
- overrides safety
- increases cognitive load
- slows session start
- collapses architecture boundaries
- introduces explanation-heavy UX
- creates config drift from M6

---

## 22. Success Criteria

M6.9 succeeds if:

- usable in <5 seconds
- improves session fit
- preserves base prescription
- remains invisible when unused
- integrates cleanly with M7 and M9

---

## 23. Implementation Rule

Implement as:

- one UI surface
- one deterministic refinement pass
- one bounded output object

No scope expansion allowed.

---

## 24. Final Principle

- M6 defines what is valid
- M6.9 refines what is usable
- M7 delivers it cleanly

**This order is mandatory.**
