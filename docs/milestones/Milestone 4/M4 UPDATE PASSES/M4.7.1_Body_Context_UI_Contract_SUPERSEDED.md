# M4.7.1 — Body Context UI Contract (M4.1.1)

## Status

Implementation contract for M4.7.1.

This contract defines the **user-facing Body Context editing layer**, completing the input loop for HARI.

M4.7.1 extends M4.1 by adding the missing UI required for users to create, edit, and manage persistent Body Context.

---

## 0. System Naming Insight

The system flow you defined:

Input → Safety → Reasoning → Mapping → Execution → Reassessment → Decision → Persistence → Validation → History

is called a:

**Closed-Loop Adaptive System Pipeline**

More specifically:

> **Human-in-the-loop adaptive regulation pipeline with persistent state feedback**

This is the correct model for HARI.

---

## 1. Purpose

M4.7.1 exists to:

* allow users to input long-term context
* complete the adaptive loop
* enable personalization across sessions

Without this layer:

* Body Context exists but is empty
* M4.3 weighting is weaker
* system is only session-adaptive, not history-adaptive

---

## 2. Core Principle

> Body Context is user-owned and explicitly controlled

The system must:

* never auto-write to Body Context
* never infer entries without confirmation
* allow full user control at all times

---

## 3. Required UI Surface

### Entry Point

Add to HomeScreen:

* "Body Context" button or link
* must be visible but not intrusive

---

### Screen

Create:

`BodyContextScreen`

This screen must:

* display categories
* allow editing within each category
* remain simple and calm

---

## 4. Supported Categories

Minimum categories:

* Injury or Event History
* Sensitive Regions
* Trigger Patterns
* Relief Patterns
* Symptom Spread Patterns
* Positional or Activity Sensitivities
* Session Structure Preferences
* User Notes

Each category must be:

* visually separated
* individually editable

---

## 5. Editing Capabilities

User must be able to:

* add entry
* edit entry
* delete entry
* clear category
* reset all Body Context

All actions must be explicit.

---

## 6. Input Model

Each entry must support:

### Structured fields

* category-specific fields

### Optional free text

* for nuance

---

## 7. Uncertainty Support

Each entry must allow:

* confirmed
* suspected
* unsure
* changed
* no longer true

System must not force certainty.

---

## 8. Source Tracking

Each entry must track:

* user_entered
* user_confirmed
* system_suggested (future)

---

## 9. Time Tracking

Each entry must include:

* created_at
* last_updated_at

---

## 10. Normalized Summary

System must derive structured summaries from entries.

Example:

Raw:
"Driving causes rib and jaw tightening"

Normalized:

* trigger: driving
* regions: rib, jaw
* response: tightening

HARI uses normalized data, not raw text.

---

## 11. Session Integration

At session start:

Show compact banner:

"Using saved Body Context"

Example summary:
"Right rib sensitivity, driving trigger, decompression helps"

Must include:

* view/edit option

---

## 12. Scope Rules

Body Context must NOT include:

* current session state
* temporary symptoms
* session outcomes
* system-generated patterns

Only persistent, reusable context.

---

## 13. Interaction with M4.7

Body Context is:

* independent from session persistence

Rules:

* deleting session does NOT delete Body Context
* invalidated sessions do NOT mutate Body Context

---

## 14. UX Requirements

The UI must feel:

* calm
* non-clinical
* low-friction
* progressive (not overwhelming)

Do NOT:

* present as medical form
* require full completion

---

## 15. Minimal Entry Rule

Body Context is optional.

System must work without it.

Encourage only:

* key regions
* key triggers
* key reliefs

---

## 16. Failure Modes to Prevent

### A — Hidden mutation

System edits context silently

### B — Overcollection

Too much required input

### C — Clinical intimidation

Feels like medical intake

### D — Data pollution

Session data stored as context

---

## 17. Implementation Boundaries

M4.7.1 implements:

* UI
* editing
* input structure

M4.7.1 does NOT implement:

* learning
* pattern detection
* session logic

---

## 18. Verification Checklist

* user can add/edit/delete entries
* categories render correctly
* summary appears in intake
* no forced input
* context persists across sessions

---

## 19. Final Instruction to Claude Code

Implement BodyContextScreen and Home entry point.

Ensure:

* user ownership
* structured + flexible input
* clean integration with intake

Do not overbuild.
Do not introduce hidden behavior.

This layer completes the adaptive loop of HARI.
