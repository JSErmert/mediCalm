# M4.7.M1 — Validated Session Read Guard Contract

## Status
Implementation contract for M4.7 Missing Item 1.

This contract adds a defensive read boundary so future history consumers cannot accidentally use invalidated sessions for adaptive decisions.

This is a small but critical safeguard that must be completed before any M5 layer reads session history for personalization or pattern learning.

---

## 1. Purpose

M4.7 persists HARI session intelligence correctly.

However, future callers could still read session history directly and accidentally include:
- invalidated sessions
- excluded sessions
- unresolved pending sessions

This contract exists to ensure:

> adaptive history reads must only consume validation-approved sessions by default

---

## 2. Core Principle

> Persistence is not the same as decision eligibility.

A session may exist in history for:
- audit
- review
- debugging
- user visibility

but still be ineligible for:
- personalization
- pattern learning
- adaptive weighting
- future decision support

---

## 3. Required Read Boundary

Create an explicit read boundary for HARI session history.

Do NOT allow future adaptive layers to read raw session history directly by default.

Instead, provide dedicated read helpers such as:

```ts
getEligibleHariHistory()
getEligibleValidatedHariSessions()

These helpers must be the default source for any future adaptive history reads.


---

## 4. Eligibility Rule

A session is eligible for adaptive reads only if:

```ts
session.session_type === "HARI"
&& session.validation_status indicates approved retention

Use the current implementation’s actual validation terminology.

If implementation currently uses:

validated
invalidated

then eligibility means:

include validated
exclude invalidated
exclude pending

If terminology is later normalized to:

kept
excluded

then the same semantic rule applies.

5. Exclusion Rule

The read boundary must exclude sessions that are:

invalidated / excluded
pending validation
non-HARI legacy sessions
malformed entries with missing required HARI metadata

This exclusion must happen automatically at the read helper level.

6. Default Behavior Rule

Future adaptive layers must not need to remember to filter manually.

The safe default must be:

read only eligible validated HARI sessions

Unsafe raw history access should not be the standard path for adaptive logic.

7. Legacy Compatibility

Legacy sessions must still remain:

visible in history UI
auditable
renderable

But they must not be treated as HARI adaptive history.

The guard must preserve:

backward compatibility
UI history rendering
audit visibility

without allowing legacy sessions into HARI decision reads.

8. Read Modes

Support two distinct history read modes:

A. Audit / Display Mode

Used for:

history screens
debugging
review
user inspection

May include:

HARI sessions
legacy sessions
validated
invalidated
pending
B. Adaptive Decision Mode

Used for:

future personalization
future pattern learning
validated session weighting
adaptive memory reads

Must include only:

validated eligible HARI sessions

This distinction must remain explicit.

9. Minimal First-Pass Implementation

For this pass, implement only the guard layer.

Do NOT introduce:

pattern learning
personalization logic
new weighting algorithms

This contract only creates the safe read boundary that future layers will use.

10. Suggested API Shape

Examples of acceptable read helpers:

getEligibleHariHistory(): HistoryEntry[]
getValidatedHariSessions(): HistoryEntry[]
isEligibleHariSession(entry: HistoryEntry): boolean

The exact naming may vary, but the semantics must be explicit and safe.

11. Failure Modes to Prevent
A — Invalidated session leakage

An excluded session influences adaptive logic.

B — Pending session leakage

An unresolved session is treated as stable evidence.

C — Legacy contamination

Legacy sessions are mixed into HARI learning inputs.

D — Manual filtering dependency

Future developers must remember to hand-filter every time.

All four must be prevented.

12. Constraints

Do NOT:

change persistence schema
redesign HistoryEntry
alter session validation behavior
implement M5 early

ONLY:

add the safe read boundary
ensure eligible-session filtering is explicit and reusable
13. Verification Checklist

After implementation:

validated HARI sessions are returned by adaptive helper
invalidated sessions are excluded
pending sessions are excluded
legacy sessions are excluded
history UI still renders all sessions normally
no persistence behavior changes
14. Final Instruction to Claude Code

Implement the validated session read guard now.

This is a defensive boundary layer, not a new learning system.

Make future adaptive history reads safe by default.

Do not expand scope beyond this guard.
