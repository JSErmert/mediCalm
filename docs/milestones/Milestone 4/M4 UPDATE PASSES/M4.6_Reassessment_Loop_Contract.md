M4.6 — Reassessment Loop Contract
Status

Implementation contract for M4.6. This contract defines the reassessment and adaptive continuation layer for HARI.

M4.6 establishes how mediCalm determines round length, when reassessment occurs, how continuation decisions are made, and how sessions adapt after the first intervention round.

This layer builds directly on M4.2 through M4.5. If later prompts conflict with this contract, this contract wins unless a higher safety rule overrides it.

1. Purpose

M4.6 defines the Reassessment Loop.

Its purpose is to make HARI sessions adaptive over time rather than one-shot.

M4.6 answers questions such as:

how many breaths should occur before reassessment?
when is there enough exposure to meaningfully test effect?
what should happen if the user feels better, the same, worse, or unclear?
when should the session continue, soften, shorten, pause, or stop?

This layer exists because HARI should not select one intervention and then assume it remains appropriate without testing the response.

2. Core Definition

The Reassessment Loop is:

current-session-focused
adaptive
conservative
response-driven
breath-round-based
safety-constrained

The Reassessment Loop is not:

an endless breathing loop
a fixed-session timer only
a blind persistence mechanism
a reward-for-endurance system

Its job is to create meaningful rounds of intervention followed by reassessment, then adjust the session accordingly.

3. Fundamental Runtime Principle

HARI must not use tiny micro-checkpoint loops that are too short to meaningfully test effect.

At the same time, HARI must not run long blind sequences without reassessment.

Therefore M4.6 must use:

meaningful breath rounds before reassessment

This is the central principle of the reassessment loop.

4. Round-Based Model

A session under M4.6 is composed of rounds, not a single uninterrupted sequence.

Each round contains:

a selected intervention package from M4.5
a defined breath count
guided execution
reassessment
continuation decision

This means:

breath count is a round property
reassessment happens after a completed round
the next step depends on response
5. Meaningful Round Length Rule

M4.6 must use larger, meaningful reassessment rounds rather than very short micro-checkpoint loops.

First-pass default round sizes
High flare / high uncertainty / fragile state → 10 breaths
Moderate / standard testing state → 20 breaths
Stable / deeper regulation state → 30 breaths

These are:

round lengths before reassessment
not guaranteed full session lengths
not proof that longer is always better

The purpose of these sizes is to allow enough exposure for a meaningful regulatory signal to emerge.

6. Why Round Sizes Are Larger

M4.6 assumes that rounds that are too short can produce false negative or low-quality readings.

Very short loops may:

stop the user before an effect appears
make the system feel artificial or overly interruptive
create measurement noise
reduce trust in the session

Therefore first-pass HARI must allow the intervention enough time to produce a testable effect before reassessment.

7. Round Length Selection Rule

Round size must be chosen using the current state estimate and intervention package, not by a fixed rule alone.

The first-pass default mapping is:

high flare / high uncertainty / high fragility → 10 breaths
moderate / mixed / standard session → 20 breaths
stable / deeper regulation / better tolerance → 30 breaths

This selection may be influenced by:

Flare_Sensitivity_Estimate
Session_Tolerance
Reassessment_Urgency
Intervention_Softness_Need
session intent
current response trend

These defaults are authoritative for first pass unless a stricter safety rule requires shorter handling.

8. Safety Interpretation of 10 / 20 / 30

M4.6 must not interpret larger round counts as permission for more aggressive intervention.

A 10-breath high-flare round means:

enough breaths to meaningfully test effect
still conservative pacing
immediate reassessment after the round

A 30-breath stable round means:

more room for a fuller regulatory arc
only when the current state supports it
still subject to reassessment and session limits

Longer rounds are not more forceful by default. They are simply longer opportunities for response to emerge.

9. Reassessment Timing Rule

Reassessment should occur after each completed round.

Do not interrupt the user with frequent mini-checkpoints that prevent meaningful testing. Do not defer reassessment indefinitely.

The standard first-pass rule is:

one full round, then reassess

This is the default runtime rhythm.

10. Reassessment Inputs

After each round, M4.6 should collect a simple current-response signal.

Minimum supported response states:

better
same
worse
mixed
unclear

Where useful, the reassessment may also note:

easier breathing
less tension
more spread
more irritation
softer or more braced

But the first-pass response logic must remain compact and usable.

11. Better Response Rule

If the user reports better after a round, the reassessment loop may:

continue the same intervention
cautiously extend the session
preserve the same softness level first
allow another round if session limits permit

Improvement is not permission for immediate escalation.

Preferred behavior:

confirm the effect is real
continue conservatively
reassess again after the next round
12. Same Response Rule

If the user reports same, the reassessment loop should:

avoid pretending the round clearly worked
consider continuing only if the state still supports it
keep the next round conservative
consider adjusting framing or intervention class if repeated sameness occurs

Same is not failure by itself. But same is also not success.

13. Worse Response Rule

If the user reports worse, the reassessment loop must not intensify.

Instead it should generally:

downshift the intervention
shorten or soften the next step
reduce performance demand
consider pause or stop
prevent escalation

This rule is mandatory.

Worsening must never be interpreted as a signal to push harder.

14. Mixed or Unclear Response Rule

If the user reports mixed or unclear, the reassessment loop should bias toward:

simpler next steps
conservative continuation if any
shorter assumption horizon
softer framing
faster reconsideration after the next step

Ambiguous response must reduce interpretive confidence.

15. Continuation Decision Set

After reassessment, M4.6 must choose one of a small number of continuation actions.

Supported continuation actions:

Continue same intervention
Continue with softer framing
Switch intervention class
Shorten next round
Pause
Stop session

These actions should be driven by:

the reassessment result
current state estimate
confidence level
session limits
16. Session Limit Rule

M4.6 must prevent open-ended sessions.

The reassessment loop must include a session ceiling.

First-pass session limit guidance

At minimum, enforce one or both of:

a maximum number of rounds
a maximum total breath count

Suggested first-pass ceiling:

generally no more than 3 rounds unless later explicitly expanded

This keeps the session bounded while preserving meaningful testing.

17. Example First-Pass Limits

Examples of acceptable first-pass ceilings:

high flare: 10 breaths × up to 2–3 rounds
moderate: 20 breaths × up to 2–3 rounds
stable: 30 breaths × up to 2 rounds by default

These are examples, not immutable medical rules.

The exact ceiling may be implemented via:

max rounds
max breaths
or both

But the session must remain bounded.

18. Total Session Logic

A session is complete when any of the following occurs:

session goal has been met sufficiently
reassessment indicates stop or hold is more appropriate
worsening persists
session limit is reached
the user chooses to stop

M4.6 must not assume that all sessions should continue until the maximum allowed round count is used.

19. Relation to M4.5

M4.5 selects the initial intervention package. M4.6 governs what happens after that package is executed for one round.

This means:

M4.5 decides the initial next step
M4.6 decides whether and how the session continues

That boundary must remain clear.

20. Runtime Compatibility with M3

M4.6 must remain compatible with the existing M3 runtime shell.

That means:

M3 still handles breath execution and visual delivery
M4.6 determines round length and reassessment cadence
M4.6 feeds M3 the current round plan
after the round, reassessment drives the next decision

M4.6 must not unnecessarily replace the breathing runtime. It should direct it.

21. Round Plan Shape

M4.6 should produce a structured round plan.

Suggested fields:

Round_Number
Breath_Count
Intervention_Class
Softness_Level
Reassessment_Required
Continuation_Constraint

Example:

Round_Number: 1
Breath_Count: 10
Intervention_Class: Soft Decompression Breathing
Softness_Level: high
Reassessment_Required: true
Continuation_Constraint: no escalation until reassessment

This structure makes runtime behavior easier to enforce.

22. Reassessment Result Shape

M4.6 should also preserve a structured reassessment result.

Suggested fields:

Round_Number
Response_Label
Response_Confidence
Next_Action
Notes_Optional

This allows the session to remain auditable and adaptive.

23. Confidence Interaction Rule

If overall session confidence is lower, M4.6 should:

favor shorter assumption horizons
avoid overinterpreting one round
reduce escalation
prefer simpler continuation decisions

Lower confidence does not automatically mean fewer than 10 breaths. It means the next decision after reassessment must remain more conservative.

24. No Endless Loop Rule

The reassessment loop must not become self-perpetuating.

Do not allow:

infinite rounds
indefinite "one more round" behavior without clear justification
loops that continue without meaningful new information

The loop exists to improve adaptation, not to create endless runtime.

25. User Experience Rule

The reassessment loop must feel supportive, not interruptive.

This means:

no excessive checkpoints
no over-frequent questioning
no feeling of being tested every few breaths
no artificial premature session endings

The user should feel that the system gave the intervention enough time to work before asking for feedback.

26. Explainability Rule

M4.6 should support explanation language such as:

"We’ll do 10 breaths first, then check how your system responds."
"This round will be longer so there’s enough time to see whether it helps."
"You felt better, so we’ll continue one more round before deciding whether to stop."

This explanation must remain:

calm
simple
non-diagnostic
practical
27. Failure Modes to Prevent

The implementation must actively guard against the following:

Failure Mode A — micro-checkpoint theater

Reassessing so quickly that the intervention never has time to work.

Failure Mode B — blind overrun

Running long sequences without reassessment.

Failure Mode C — worsening escalation

Treating worse as a reason to push harder.

Failure Mode D — endless continuation

Letting the session continue indefinitely.

Failure Mode E — false success bias

Treating "same" or "unclear" as clear success.

Failure Mode F — interruption fatigue

Asking so often that the session becomes irritating.

All six must be prevented.

28. Implementation Boundaries

M4.6 implements:

round-length logic
reassessment timing
response handling
continuation decisions
bounded session progression

M4.6 does not implement:

Body Context storage
initial state estimation itself
link mapping itself
initial intervention selection itself
advanced long-term learning

Those belong to earlier or later layers.

29. Final Instruction to Claude Code

Implement M4.6 as the reassessment and adaptive continuation layer for HARI.

Your implementation must:

use meaningful breath rounds before reassessment
default to first-pass round sizes of 10 / 20 / 30 breaths depending on current state
treat those counts as round lengths, not guaranteed full sessions
reassess after each round
continue conservatively based on response
prevent endless sessions
remain compatible with the M3 breathing runtime

Do not revert to tiny checkpoint loops. Do not treat longer rounds as permission for more aggression. Do not let worsening intensify the session.

This layer must make mediCalm feel meaningfully testable, adaptive, and trustworthy.
