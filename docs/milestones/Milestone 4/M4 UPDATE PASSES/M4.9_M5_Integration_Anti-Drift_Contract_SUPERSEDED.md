## 2. Carried-Forward Technical Debts

These items were documented during M4 and explicitly deferred. M5 must resolve or formally extend each one before building on top of it.

### D1 — `round_plan` singular vs. plural (Priority: HIGH before M5 adaptive reads)
**Location:** `src/types/hari.ts` — `PersistedHariMetadata.round_plan: RoundPlan`
**Issue:** M4.7 contract §3 specifies `round_plan: RoundPlan[]` (array, supports multi-round history). Current type stores only a single `RoundPlan`.
**Risk:** Any M5 logic that reads `hari_metadata.round_plan` expecting an array will fail silently or require a shape check. Existing saved sessions all have the singular shape.
**Required action before M5 reads this field:** Write a migration spec. Either rename field + version the shape, or write a backwards-compatible transformer before M5 pattern learning touches it.

### D2 — `user_selected_action: 'exit'` never written (Priority: LOW)
**Location:** `src/types/hari.ts` — `UserSelectedAction = 'continue' | 'finish' | 'exit'`
**Issue:** The `'exit'` variant is defined but never written to a session record. Discard path calls `CLEAR_SESSION` without persisting.
**Risk:** M5 pattern learning expecting to find `'exit'` records in history will find none. Not a bug — discarded sessions are never saved — but the type implies a capability that doesn't exist.
**Required action:** Either remove `'exit'` from the type or document that discarded sessions are intentionally absent from history.

### D3 — Exit gating during reassessment (Priority: MEDIUM — F1 from M4.8)
**Location:** `GuidedSessionScreen.tsx` — reassessment response block
**Issue:** "Exit & discard" is only reachable after the user selects a reassessment response. M4.6.1 §5 requires it to always be available.
**Risk:** User cannot immediately exit during reassessment without first selecting a response. Not a safety failure (safety interrupt is always available), but a UX integrity gap.
**Required action:** M5 should resolve this. "Exit & discard" button must render unconditionally in the reassessment phase.

### D4 — `pain_input` screen deprecated but not removed (Priority: LOW)
**Location:** `App.tsx:35`, `AppContext.tsx:11`
**Issue:** `'pain_input'` remains in the AppScreen type and router. Nothing in M4 production flow navigates to it. Retained because `AppContext.test.tsx` tests it.
**Required action:** Remove from production router in M5 cleanup pass. Update or delete the AppContext test that navigates to it.

### D5 — `HariSafetyGateScreen` render-time dispatch (Priority: LOW)
**Location:** `src/screens/HariSafetyGateScreen.tsx`
**Issue:** Guard fires during render (`if (!intake) dispatch(...)`) rather than in a `useEffect`. Works in practice; fragile under React Strict Mode / concurrent mode.
**Required action:** Convert to `useEffect` guard before M5 adds complexity around that screen.

## 3. M5 Prerequisites

M5 must NOT begin until each prerequisite is satisfied.

### P1 — D1 resolved
`round_plan` migration spec written and accepted. No M5 adaptive read logic may touch `hari_metadata` until the shape contract is confirmed.

### P2 — D3 resolved
Exit gating fixed. M5 should not add reassessment complexity on top of a known UX integrity gap.

### P3 — Adaptive read boundary confirmed
Any M5 layer that reads session history for personalization MUST use `getEligibleHariHistory()` (M4.7.M1). Never `loadHistory()`. This is a hard architectural rule, not a suggestion.
If M5 adds a new adaptive read path, it must be reviewed against M4.7.M1 before shipping.

### P4 — Body Context schema confirmed stable
M5 must not silently extend `BodyContextCategory` or `BodyContextItem` without a schema version bump. The current 8-category model is the stable M4 baseline.

### P5 — Safety gate authority preserved
`HariSafetyGateScreen` is the sole authority for deciding whether a session may proceed. M5 must not add a secondary gate, bypass it, or make it conditional on adaptive state. If M5 needs to influence session entry, it does so by pre-populating intake fields — never by replacing or wrapping the gate.

## 4. Safe Boundaries for M5

### What M5 MAY do
- Read `getEligibleHariHistory()` for pattern detection
- Use validated HARI session history to inform intake defaults or protocol hints
- Extend `PersistedHariMetadata` with new optional fields (must be backward-compatible — no required fields added without migration)
- Add new adaptive fields to `HariSessionIntake` as optional (existing required fields must not be removed)
- Display trend data or session summaries derived from validated history
- Introduce new screen surfaces (e.g., a history insights screen) without altering existing production flow
- Extend `protocolSelection` with adaptive weight adjustments, gated behind `hariHintProtocolId` (already the designated hook)

### What M5 MUST NOT do
- Read `loadHistory()` for adaptive decisions
- Use `pending` or `invalidated` sessions as adaptive inputs
- Make intake fields optional that are currently required
- Remove or weaken the validation gate
- Store adaptive state in Body Context (wrong truth class — D is derived, A is user-managed)
- Mix truth classes A/B/C/D (M4 separation rule, non-negotiable)
- Auto-advance the user based on adaptive inference without explicit confirmation
- Override safety-critical routing based on pattern history

## 5. M5 Proposition

Based on M4 reality, M5 should be scoped as follows. This is a proposition, not a contract. It requires a dedicated M5.0 Identity Contract before any implementation begins.

### Proposed M5 Identity: Adaptive Regulation Layer

**Core thesis:** M5 uses validated HARI session history to make the intake and protocol selection experience progressively more calibrated — without removing user agency, weakening safety, or introducing personalization hype.

**Proposed scope:**

#### M5.1 — Session Pattern Reader
Read `getEligibleHariHistory()` and derive lightweight signals:
- Dominant symptom focus across recent sessions
- Most-used protocol IDs
- Average pain trajectory (before → after delta)
- Flare sensitivity trend

No UI in M5.1. This is a pure data layer — a `getSessionPatterns()` function, tested in isolation.

#### M5.2 — Adaptive Intake Defaults
Use session patterns to pre-populate intake fields as soft defaults (user can override any of them):
- Pre-select `symptom_focus` if strongly dominant
- Pre-select `flare_sensitivity` if consistent
- Pre-select `session_intent` if user tends toward one type

Soft defaults only. Never locked. Clearly marked as suggested.

#### M5.3 — Protocol Hint Reinforcement
Pass the dominant protocol hint through the existing `hariHintProtocolId` hook in `protocolSelection.ts`. The engine already supports this; M5.3 just provides a data-driven input.
Safety blocks remain unchanged — the hint is still blocked if contraindicated.

#### M5.4 — Session Insights Surface (optional, defer if scope grows)
A lightweight history summary view: trend line, protocol frequency, average pain delta. Read-only. No gamification. No streaks. No badges.

### Proposed M5 non-scope
- Predictive session scheduling
- Notifications or reminders
- Cloud sync or account system
- Social or comparative features
- Symptom diagnosis or medical claims
- Automated session selection without user confirmation

## 6. Anti-Drift Rules for M5

These rules exist because adaptive systems tend to drift toward overreach. They are non-negotiable regardless of how M5 is implemented.

**Rule 1 — Adaptive inference is advisory only.**
M5 may suggest. It may not decide. The user confirms every session start.

**Rule 2 — History eligibility is binary.**
A session is either fully eligible (HARI + validated + metadata present) or excluded. No partial credit, no fallback weighting.

**Rule 3 — Safety outranks personalization.**
If adaptive data and safety rules conflict, safety wins. Always.

**Rule 4 — No silent adaptation.**
If intake fields are pre-populated from history, the user must be able to see and override them. No invisible personalization.

**Rule 5 — Pattern learning requires a minimum sample.**
Do not apply adaptive defaults until at least N validated sessions exist (N to be defined in M5.1 contract — suggested minimum: 3). Below the threshold, system behaves as M4 baseline.

**Rule 6 — Degradation must be graceful.**
If `getEligibleHariHistory()` returns an empty array (new user, all sessions invalidated, storage cleared), the app must behave exactly as it did in M4. No errors, no broken defaults, no missing screens.
