# M5.2 — Adaptive Intake Defaults Contract

## Status
Definition layer for adaptive intake behavior.

This contract defines how HARI may use M5.1 PatternSummary to influence session intake defaults.

No direct user override removal is permitted.
No hidden behavior is allowed.

---

## 1. Purpose

M5.2 exists to:

- reduce friction at session start
- improve initial alignment with user tendencies
- apply safe, bounded personalization

M5.2 does NOT:
- replace user input
- lock decisions
- infer new data
- modify system behavior beyond allowed fields

---

## 2. Core Principle

> M5.2 may suggest defaults, never enforce them.

All adaptive behavior must be:
- visible
- reversible
- optional
- safety-constrained

---

## 3. Input Source

M5.2 reads only from:

```ts
getOrComputePatternSummary()

No raw history access is permitted.

4. Eligibility Rule

A pattern may influence intake ONLY if:

state ∈ { 'active', 'high_confidence' }
state ≠ 'suspended'
no safety contradiction is active

Otherwise:

ignore the pattern completely
5. Allowed Personalization Fields

M5.2 may only influence:

5.1 Session Intent
suggest most common validated intent
5.2 Symptom Focus
suggest most common selected focus
5.3 Flare Sensitivity (⚠️ constrained)
may suggest higher sensitivity
must NEVER suggest lower than recent validated trend
5.4 Session Length Preference
suggest common preference
6. Disallowed Fields

M5.2 must NOT influence:

safety gate behavior
intervention class selection
protocol mapping
body context
persistence logic
validation rules
7. Application Behavior
7.1 Pre-Population Only

Defaults appear as:

pre-filled selections
visually indicated as suggestions

User must be able to:

change any field
ignore suggestions
proceed without interaction penalty
7.2 Visibility Requirement

Suggested defaults must be visibly marked.

Examples:

subtle highlight
label: "Suggested"
soft visual cue
7.3 No Lock-In

User changes must:

override suggestion immediately
not be resisted
not trigger correction loops
8. Safety Constraints
8.1 Flare Sensitivity Rule

If recent pattern includes:

'high' flare sensitivity

Then:

suggestion must NOT default to 'low'
may default to 'moderate' or 'high' only
8.2 Contradiction Rule

If pattern has:

recent contradiction
safety contradiction
suspended state

Then:

do NOT apply any suggestion
8.3 Uncertainty Rule

If pattern state is:

'candidate'
'insufficient'

Then:

no default may be applied
9. Explainability

The system must support:

"Based on your recent sessions, this is suggested — you can change it anytime."

No hidden personalization allowed.

Each suggested field must carry its own pattern_state so any future surface can
explain which confidence level drove that specific suggestion independently.
A field with pattern_state 'high_confidence' may be presented more prominently
than one with state 'active', but both remain fully overridable by the user.

10. Failure Modes to Prevent
A — Hidden personalization

User unaware defaults are influenced

B — Overconfidence

Weak patterns used as strong defaults

C — Safety violation

Lowering sensitivity despite recent high flare

D — Lock-in behavior

User cannot easily override suggestion

E — Cross-field contamination

One pattern influencing unrelated fields

11. Graceful Degradation

If:

PatternSummary is null
no active patterns exist
computation fails

Then:

fallback to standard M4 intake defaults
no personalization applied
12. Output Shape

M5.2 must produce:

```ts
interface AdaptiveIntakeDefaults {
  session_intent?:            { value: SessionIntent;            pattern_state: PatternState }
  symptom_focus?:             { value: SymptomFocus;             pattern_state: PatternState }
  flare_sensitivity?:         { value: FlareSensitivity;         pattern_state: PatternState }
  session_length_preference?: { value: SessionLengthPreference;  pattern_state: PatternState }
}
```

Rules:

- A field is present only when its source dimension is 'active' or 'high_confidence'.
- A field being absent (undefined) means no eligible pattern — no default is applied for that field.
- Each field carries its own pattern_state independently. Fields are not coupled.
- No field may be populated from a 'candidate', 'suspended', or 'insufficient' pattern.
- The flare_sensitivity field must never carry a value lower than the most recent
  validated flare_sensitivity observation, regardless of pattern_state.
  Authority: M5.2 §8.1, M5.1 §5.6

This shape replaces the prior single source_pattern_state field. Rationale: each
intake field is derived from an independent DetectedPattern at a potentially
different confidence level. A shared state was unexplainable and would require an
arbitrary tie-break rule. Per-field state is required by M5.0 §9.4 and M5.1 §9.4.
13. Integration Rules

M5.2 must:

run before intake render
NOT modify stored data
NOT persist changes
NOT alter validation logic
14. Final Instruction to Claude Code

Implement M5.2 as a pre-population layer for intake.

Do NOT:

override user input

Ensure:

all personalization is visible

all behavior is reversible
