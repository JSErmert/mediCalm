# M5.6 — Codebase Architecture & Organization
## Design Spec

**Date:** 2026-03-31
**Status:** Approved — ready for implementation planning
**Milestone:** M5.6
**Type:** Structure-only refactor

---

## 1. Purpose

M5.6 reorganizes the source tree to improve file discoverability, naming clarity, and structural coherence.

It does not change any logic, behavior, UI, or data flow.

Every file that moves or is renamed must have its import paths updated in the same pass.

---

## 2. Scope Boundaries

### Permitted

- Move files to new paths
- Rename files
- Update import paths broken by moves/renames
- Create new subdirectories as needed

### Prohibited

- Logic changes of any kind
- Behavior changes of any kind
- UI changes of any kind
- Feature additions
- File deletions
- Modifications to `types/index.ts` content
- Modifications to `engine/hari/systemDefinition.ts` content
- Import path changes beyond those required by the moves

---

## 3. Structural Problems Being Fixed

### 3.1 — `engine/` mixes session pipeline with presentation utilities

The M3 session pipeline (`safetyPrecheck`, `mechanismScoring`, `protocolSelection`, `sessionBuilder`) is a coherent execution unit. Three files in `engine/` serve a different role — they are presentation utilities that consume session/engine output to produce UI parameters and strings:

- `expressionProfile.ts` — derives `ExpressionProfile` (orb scale, glow, microtext) from `RuntimeSession`
- `interpretationLayer.ts` — derives focus text and breathing hint shown on setup screen and case file
- `microGuidance.ts` — generates per-breath-phase guidance strings for the orb

These files belong downstream of the engine, not inside it. Grouping them in `engine/presentation/` makes the distinction explicit.

### 3.2 — Dev utility inside `engine/`

`engine/devOverride.ts` is called exclusively by `RDReviewScreen` to build a session while bypassing the safety gate. It has no runtime role and does not belong in the session pipeline folder. It belongs with `utils/devFlags.ts`, the other dev-only utility.

### 3.3 — `bodyContext` name collision across layers

Two files share the basename `bodyContext` at different layers:

| Path | Role |
|------|------|
| `src/storage/bodyContext.ts` | Persists and loads raw `BodyContext` to/from localStorage |
| `src/engine/hari/bodyContext.ts` | Transforms raw `BodyContext` into a normalized `BodyContextSummary` for the HARI engine |

The engine file exports `buildBodyContextSummary()`. After rename to `bodyContextSummary.ts`, the filename matches the function's role and the collision is resolved.

## 6. Types Compatibility Layer

`src/types/index.ts` contains a re-export block at the bottom that re-exports HARI types from `./hari`:

```ts
export type {
  HariSessionIntake,
  SessionIntent,
  // ...
} from './hari'
```

**Finding:** No consumer imports HARI types via `../types`. All HARI type consumers import directly from `../types/hari`. The re-export block is dead code.

**Decision:** `types/index.ts` is treated as a compatibility layer and is left unchanged. It will not be modified in M5.6. Any future cleanup of this file is a separate decision requiring separate verification.

---

## 7. Architectural Anchors — Files Not Moved

### `src/engine/hari/systemDefinition.ts`

This file contains the M4.0 philosophy and constraint lock for HARI. It is not imported by any module at runtime. It is referenced only in a comment in `engine/hari/index.ts`.

**Decision:** Treated as an architectural anchor. It documents invariants that no later module may weaken. It stays in place and is not moved or modified.

## 9. Verification Requirements

Every move and rename must be verified before the implementation pass is considered complete.

### Required verification steps

1. **TypeScript build** — `tsc --noEmit` must pass with zero errors
2. **Full test run** — `vitest run` must pass: 20 test files, 181 tests, 0 failures
3. **Import audit** — after each move group, grep for the old path to confirm no import sites remain

### Import audit commands (run after each group)

```bash
# Group 1 — confirm no stale presentation imports
grep -rn "from.*engine/expressionProfile\|from.*engine/microGuidance\|from.*engine/interpretationLayer" src/

# Group 2 — confirm devOverride moved cleanly
grep -rn "from.*engine/devOverride" src/

# Group 3 — confirm bodyContext rename complete
grep -rn "from.*engine/hari/bodyContext[^S]" src/
```

---

## 10. Execution Rules

1. Implement one group at a time (Group 1, then Group 2, then Group 3).
2. Run the import audit grep after each group before proceeding.
3. Run the full verification suite (tsc + vitest) only after all three groups are complete.
4. If any verification step fails, stop and resolve before continuing.
5. Do not combine groups into a single bulk move.
6. Do not add files beyond what is specified.
7. Do not modify file content beyond import path corrections.
