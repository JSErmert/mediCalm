# mediCalm AlignFlow Governance v1.0

## Document Type
Internal governance architecture

## Project
mediCalm

## System
AlignFlow (adapted)

## Purpose
This document governs how mediCalm expands without outrunning its structural integrity.

It exists to ensure that:
- milestones emerge in the correct order
- foundational layers stabilize before higher-order layers expand
- scope does not broaden faster than the system can safely support
- new features are placed according to dependency logic, not enthusiasm

## Core Principle

> mediCalm must not grow faster than its architecture can safely hold.

---

## 1. What AlignFlow Governs in mediCalm

AlignFlow governs:

- milestone order
- dependency sequencing
- readiness thresholds
- stabilization before expansion
- scope containment
- structural maturity decisions
- what is foundational vs premature

It does not govern:
- runtime session logic
- user-facing breathing prescriptions
- copywriting specifics
- detailed implementation mechanics

---

## 2. mediCalm Structural Layering

The canonical progression is:

- M6 — WHAT TO DO
- M7 — HOW TO DELIVER
- M6.9 — bounded post-prescription refinement (implemented only after M7.3)
- M8 — LEARNING
- M9 — FAST ENTRY
- M10 — DAILY INTEGRATION

No new feature may be inserted arbitrarily if it violates this sequence.

---

## 3. Foundational Rule

A proposed addition must answer:

1. What layer does this belong to?
2. What existing layer does it depend on?
3. What must already be stable before it is added?
4. Is this foundational, or is it premature?
5. Does this increase calm precision, or just increase system size?

If these questions cannot be answered clearly, the feature is not ready.

---

## 4. Readiness Thresholds

A milestone or sub-layer is considered ready to expand only when:

- its core purpose is clearly defined
- boundaries are stable
- implementation responsibilities are separated cleanly
- major contradictions are resolved
- there is a usable audit path for future drift

Examples:
- M6 had to stabilize before M7 could responsibly shape delivery
- M7 must stabilize before M6.9 can safely bias delivery
- M8 should not become aggressive learning before truth-status discipline is clear

---

## 5. Scope Containment Rules

Do not add:
- more user choice when the current layer is still unclear
- new adaptive behaviors before existing ones are bounded
- convenience surfaces that bypass feasibility or safety
- branching UX when deterministic flow is still being refined
- explanatory complexity that harms calm usability

When in doubt:
- stabilize first
- simplify first
- sequence before branching

---

## 6. Prematurity Test

A feature is premature if it:

- depends on undefined downstream behavior
- modifies a layer that has not been locked yet
- duplicates responsibility already owned elsewhere
- increases cognitive load before solving structural fit
- adds visible sophistication without foundational need

Premature work should be deferred, not force-fit.

---

## 7. Active Governance Questions

Before any milestone or feature is approved, ask:

- Is this the next correct layer?
- Does it protect the calmness of the experience?
- Does it preserve separation of concerns?
- Will this create future cleanup debt?
- Is this a structural necessity or an attractive extra?

---

## 8. Allowed Outputs of AlignFlow Review

AlignFlow review may output:

- proceed now
- defer until later milestone
- lock concept, delay implementation
- split into smaller ordered phases
- reject as structurally misaligned
- merge into an existing responsibility instead of creating a new one

---

## 9. Failure Conditions

AlignFlow governance has failed if mediCalm begins to:

- add features out of milestone order
- collapse multiple layers into one
- introduce branch-heavy UX too early
- let fast-entry logic outrun safety logic
- build adaptive behavior before delivery behavior is defined
- accumulate “smart” features that reduce calmness

---

## 10. Final Rule

If a proposal improves excitement but weakens structure, structure wins.

mediCalm should mature in an intelligible sequence:
- foundation
- delivery
- bounded refinement
- learning
- acceleration
- integration
