# M4.6.1 — Reassessment Control Contract

## Status
Implementation contract for M4.6.1.

This contract defines the **user-controlled reassessment decision layer** for HARI.

M4.6.1 extends M4.6 by ensuring that:
- reassessment results do NOT automatically trigger continuation
- the user explicitly chooses what happens next
- exit is always available
- the system remains adaptive but never forceful

This is a critical safety and experience layer.

---

## 1. Purpose

M4.6.1 exists to ensure that:

> reassessment is a decision point, not an automatic trigger

The system must:
- collect user response
- compute a recommended next step
- **wait for user confirmation before continuing**

This preserves:
- safety
- trust
- user agency
- correct nervous system behavior

---

## 2. Core Principle

> Response selection must NEVER automatically start the next round.

The system must separate:

- **Response** (Better / Same / Worse / Mixed / Not sure)
- **Action** (Continue / Finish / Exit)

These are not the same.

---

## 3. Required Flow

After each round:

### Step 1 — Reassessment Prompt

Ask:

**How does your body feel?**

Options:
- Better
- About the same
- Worse
- Mixed
- Not sure

---

### Step 2 — Store Response

When user selects a response:

- store it in reassessment result
- DO NOT start next round
- DO NOT navigate
- DO NOT auto-trigger any action

The system may compute an internal recommendation, but must not act yet.

---

### Step 3 — Present Action Choices

After response selection, show:

---

## Primary Action

### Continue session
- starts next round
- uses M4.6 logic
- adapts based on response

---

## Secondary Action

### Finish & save results
- ends session
- persists session data
- transitions to post-session screen

---

## Always Available

### Exit & discard session
- cancels session
- deletes current session data
- returns to HomeScreen

---

## 4. No Auto-Advance Rule

The following is strictly prohibited:

- starting the next round immediately after response selection
- bypassing user confirmation
- chaining rounds without explicit user input

Even if:
- response = Better
- system is confident
- session is stable

The user must still confirm continuation.

---

## 5. Exit Availability Rule

Exit must always be available:

- after each reassessment
- regardless of response
- regardless of state
- regardless of round number

Optional:
- allow exit during breathing round via "Stop" button

Exit must:
- discard session
- prevent session persistence
- return user safely to home

---

## 6. Recommendation Behavior

The system may suggest:

Examples:
- “We can continue one more round”
- “It may be better to stop here”
- “We’ll keep this shorter and softer”

But must not:
- auto-act on recommendation
- override user choice

---

## 7. UI Behavior Rules

### Response Selection

- selecting Better/Same/Worse highlights selection
- does not navigate
- does not trigger session continuation

---

### Action Buttons

Display clearly:

- Continue (primary, emphasized)
- Finish & Save (secondary)
- Exit & Discard (always visible, lower emphasis but accessible)

---

### Button Hierarchy

1. Continue session (primary CTA)
2. Finish & save (secondary)
3. Exit & discard (subtle but always present)

---

## 8. Data Model

Extend reassessment result:

```ts
{
  Round_Number,
  Response_Label,
  Response_Confidence,
  Recommended_Action,
  User_Selected_Action
}

User_Selected_Action must be explicitly recorded.

9. Integration with M4.6

M4.6 determines:

round length
recommended next action

M4.6.1 determines:

whether the system is allowed to proceed

M4.6 suggests
M4.6.1 confirms

10. Safety Rule

User control overrides system suggestion.

If user chooses:

stop → stop
exit → exit
continue → continue

System must not resist or override.

11. Failure Modes to Prevent
A — Auto-advance

System continues without confirmation

B — Hidden continuation

System advances subtly without clear user action

C — Trapped session

User cannot exit easily

D — Forced completion

User must finish session before leaving

E — Over-automation

System behaves like a script, not adaptive guidance

All must be prevented.

12. Experience Rule

The reassessment layer must feel:

calm
optional
user-led
non-pressuring
safe

The user must feel:

“I am choosing what happens next”

13. Final Instruction to Claude Code

Implement M4.6.1 as the reassessment control layer.

Your implementation must:

separate response selection from continuation
prevent automatic round progression
require explicit user action before continuing
support finish & save
support exit & discard at all times
preserve system recommendations without enforcing them

Do not:

auto-trigger next round
remove exit options
force session progression

This layer is required to make HARI safe, trustworthy, and human-centered.
