# M4.7 — Session Intelligence Persistence Contract

## Status

Implementation contract for M4.7.

This contract defines how mediCalm persists HARI session intelligence so the system can evolve from session-based adaptation to history-aware learning.

M4.7 establishes the data lifecycle for:

* HARI session metadata
* reassessment history
* intervention tracking
* baseline vs outcome comparison

This layer builds on M4.0–M4.6.1.

---

## 1. Purpose

M4.7 exists to ensure that:

> HARI decisions are not lost after a session ends.

Without M4.7:

* HARI adapts within a session
* but learns nothing across sessions

With M4.7:

* session intelligence is retained
* future layers can learn patterns
* the system becomes truly history-adaptive

---

## 2. Core Principle

> Only validated sessions contribute to persistent intelligence.

This contract must respect:

* session validation gate
* deletion horizon
* reversion rules

---

## 3. What Must Be Persisted

Each completed session must store a **HARI metadata block**.

### Required fields

```ts
hari_metadata: {
  intake: HariIntake,
  safety_result: "CLEAR" | "HOLD" | "STOP",
  state_estimate: HariStateEstimate,
  link_map: HariLink[],
  intervention: HariInterventionPackage,
  round_plan: HariRoundPlan[],
  reassessment_history: HariReassessmentResult[],
  baseline_intensity: number,
  post_session_intensity?: number
}
```

---

## 4. HistoryEntry Integration

Extend HistoryEntry:

```ts
HistoryEntry {
  id,
  created_at,
  validation_status,
  session_data,
  hari_metadata?: HariSessionMetadata,
  session_type: "HARI" | "LEGACY"
}
```

### Rules

* HARI sessions must include hari_metadata
* legacy sessions must not be forced to include it
* UI must differentiate HARI vs legacy sessions

---

## 5. Session Type Differentiation

The system must distinguish:

### HARI sessions

* have hari_metadata
* use M4 flow

### Legacy sessions

* no hari_metadata
* use M3 flow

UI must be able to:

* label or visually distinguish them
* handle both without error

---

## 6. Save Behavior Rules

### Finish & Save

* persist full hari_metadata
* mark session as pending validation

### Exit & Discard

* DO NOT persist session
* DO NOT write hari_metadata

### HOLD / STOP

* no session created
* no persistence

---

## 7. Validation Integration

Only sessions marked:

```ts
validation_status = "kept"
```

may:

* contribute to future adaptive layers
* be used in pattern detection (future M layers)

Sessions marked "excluded" must:

* remain in history for audit
* NOT influence future logic

---

## 8. Reversion Compatibility

If a kept session is later invalidated:

* its hari_metadata must be excluded from influence
* no global recalculation required (first pass)

This preserves:

* short-horizon rollback model

---

## 9. Baseline vs Outcome Tracking

Each HARI session must support:

* baseline intensity (pre-session)
* post-session intensity (if provided)

This enables:

* before/after comparison
* future effectiveness evaluation

---

## 10. Reassessment History Storage

Each round must be preserved:

```ts
{
  round_number,
  response,
  user_selected_action,
  recommended_action
}
```

This is critical for:

* pattern learning
* debugging behavior
* system explainability

---

## 11. No Silent Promotion Rule

M4.7 must NOT:

* write to Body Context
* create long-term patterns automatically
* convert session data into permanent truth

That belongs to future learning layers.

---

## 12. Failure Modes to Prevent

### A — Data loss

HARI decisions disappear after session

### B — History pollution

invalid sessions influence learning

### C — Legacy breakage

old sessions fail to render

### D — Silent mutation

system rewrites user-owned data

### E — Overcollection

storing unnecessary or unused fields

---

## 13. Implementation Boundaries

M4.7 implements:

* session metadata persistence
* history differentiation
* baseline/outcome tracking
* reassessment history storage

M4.7 does NOT implement:

* learning algorithms
* pattern detection
* Body Context editing
* UI redesign

---

## 14. Verification Requirements

After implementation:

* HARI session saves metadata correctly
* discarded session does not persist
* legacy sessions still render
* HARI vs legacy differentiation visible
* reassessment history stored
* baseline intensity stored
* invalidated sessions excluded from influence

---

## 15. Final Instruction to Claude Code

Implement M4.7 as the persistence layer for HARI.

Your implementation must:

* store complete HARI session metadata
* distinguish HARI vs legacy sessions
* respect validation and deletion rules
* preserve auditability and future compatibility

Do not:

* merge layers
* introduce learning prematurely
* break existing session rendering

This layer enables HARI to become truly history-adaptive.
