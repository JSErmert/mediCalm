# M4.8.1 — Audit Remediation Contract

## Status
Implementation contract for M4.8.1.

This contract resolves the required fixes and contained remediation items identified by the M4.8 sequential audit.

M4.8.1 is a stabilization pass.
It does not introduce new product behavior or new system layers.
It exists to close audit findings cleanly before M4 is finalized.

---

## 1. Purpose

M4.8.1 exists to:

- resolve audit-required fixes
- close accessibility gaps found during M4.8
- remove low-risk inconsistencies
- prepare M4 for formal closure

This is a remediation pass, not a redesign pass.

---

## 2. Scope

M4.8.1 implements only the following:

### RF1 — Dev banner copy fix
Correct the inverted or misleading dev banner copy.

### RF2 — Reduced motion support in GuidedSessionScreen
Add the missing reduced-motion handling in `GuidedSessionScreen.module.css`.

### RF3 — Reduced motion support in CompletionForm
Add the missing reduced-motion handling in `CompletionForm.module.css`.

### RF4 — Stale field-count comment
Correct the stale field-count comment in `SessionIntakeScreen`.

---

## 3. Optional Low-Risk Cleanup

If safe and trivial, M4.8.1 may also include:

### Dead route cleanup
Review `pain_input` route.
If truly unreachable and obsolete, remove it.
If removal is risky, explicitly mark it deprecated and leave behavior unchanged.

This is optional and must not expand into route redesign.

---

## 4. Explicit Non-Goals

M4.8.1 must NOT:

- redesign reassessment flow
- change HARI engine behavior
- change persistence schema
- implement pattern learning
- redesign Body Context UI
- start M5 early

Those belong to other layers.

---

## 5. Accessibility Rule

Reduced motion fixes must be treated as required stability work, not polish.

Where animation or transition exists on main user-facing session surfaces:
- reduced motion mode must suppress or simplify those effects
- behavior must remain usable and understandable
- no safety-critical messaging may depend on motion

---

## 6. Carry-Forward Items

M4.8.1 may document but does not need to resolve:

- M4.7 schema concern: round plan singular vs plural
- future read-boundary concerns already handled by M4.7.M1
- any M5 pattern-learning prerequisites
- accepted open item F1 if already owned by M4.7.2

These must be preserved in M4 closeout notes.

---

## 7. Verification Checklist

After implementation:

- dev banner copy is corrected
- GuidedSessionScreen reduced-motion path exists
- CompletionForm reduced-motion path exists
- stale intake comment is corrected
- no production behavior regression introduced
- no new scope added

---

## 8. Output Report Format

After completion, report:

- Files changed
- Audit fixes completed
- Optional cleanup completed or deferred
- Remaining carry-forward items for M4 closeout

Keep the report concise.

---

## 9. Final Instruction to Claude Code

Implement M4.8.1 as a strict audit remediation pass.

Resolve the required audit fixes only.

Do not redesign.
Do not expand scope.
Do not begin M5 work.

This pass exists to make M4 clean, accessible, and ready for closure.
