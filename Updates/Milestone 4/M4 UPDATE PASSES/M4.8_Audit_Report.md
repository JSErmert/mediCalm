## Fail

### F1 ‚Äî Exit & discard gated during reassessment (known open item)
**Location:** `GuidedSessionScreen.tsx` ‚Äî reassessment response selection block  
**Symptom:** "Exit & discard" action is conditionally available only after the user selects a reassessment response option. Contract M4.6.1 ¬ß5 requires this action to always be available, independent of response selection.  
**Status:** User acknowledged and accepted current state. Remains open for M4.9+.

### F2 ‚Äî Dev banner copy inverted
**Location:** `HomeScreen.tsx` ‚Äî dev banner conditional  
**Symptom:** Banner text reads "blocks route to review screen" when `devMode` is ON. Logic is inverted ‚Äî dev mode *enables* the override route, not blocks it. Misleading during development but no production impact (banner not shown in production).

## Missing

### M1 ‚Äî `user_selected_action: 'exit'` never recorded
`UserSelectedAction` type includes `'exit'` but the discard path calls `CLEAR_SESSION` + navigate without writing this value to the session record. Moot since discarded sessions are never saved, but the type is incomplete.

### M2 ‚Äî `PersistedHariMetadata.round_plan` stores singular not array
M4.7 contract ¬ß3 specifies `round_plan: RoundPlan[]` (plural, supporting multi-round history). Current type in `src/types/hari.ts` stores `round_plan: RoundPlan` (singular ‚Äî only current round). Adaptive replay of multi-round sessions will require a migration when this is expanded in M5+.

### M3 ‚Äî `pain_input` route unreachable but still declared
`'pain_input'` is listed in `AppContext` screen union type and the `App.tsx` router but nothing in the M4 production flow navigates to it. Only present in `AppContext.test.tsx`. Not a production risk, but dead surface.

## Regression Risks

**R1 ‚Äî `PersistedHariMetadata.round_plan` singular vs. plural (‚Üí M3 above)**  
If M5 adaptive logic is written expecting `round_plan[]`, it will break silently against existing saved sessions. Needs a migration spec before M5 personalization work begins.

**R2 ‚Äî HariSafetyGateScreen render-time dispatch**  
`if (!intake) { dispatch({ type: 'NAVIGATE', screen: 'session_intake' }) }` fires during render rather than in an effect. Works in practice (React batches the bailout) but is fragile ‚Äî side effects during render can double-fire under concurrent mode or Strict Mode. Low immediate risk, worth a `useEffect` conversion before M5.

**R3 ‚Äî GuidedSessionScreen.module.css hardcoded transitions not reduced-motion guarded**  
`opacity` transitions (0.15s‚Äì0.2s) on phase transitions and completion form appearance are hardcoded inline, not using `--transition-quick`. The global token fallback in `tokens.css` does not cover these. Users with `prefers-reduced-motion: reduce` will still see motion on the main session surface. Not a safety failure but an accessibility gap.

**R4 ‚Äî CompletionForm.module.css hardcoded transitions**  
`.resultButton`, `.noteInput`, `.saveButton`, `.discardButton` all use hardcoded `transition: ... 0.15s ease`. Not covered by `--transition-quick` token fallback. Same class as R3, lower severity (non-animated transitions).

**R5 ‚Äî SessionIntakeScreen comment/field count mismatch**  
Comment says "5 fields" but `baseline_intensity` was added via M4.5.2 patch, making 6. Submit guard `allSelected` is correct (only 5 required). Comment is stale and could mislead future maintainers.
