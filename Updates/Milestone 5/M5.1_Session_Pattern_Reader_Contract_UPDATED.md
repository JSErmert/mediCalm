# M5.1 — Session Pattern Reader Contract

## Status
Definition layer only.
No implementation authorized by this contract.
No UI authorized by this contract.
No M4 modification authorized by this contract.

Parent authority: M5.0 Adaptive Intelligence Identity Contract.

---

## 1. Purpose

M5.1 defines how validated HARI session history is read, analyzed, and converted into bounded pattern signals for downstream adaptive use.

M5.1 is a pure data layer. It produces a `PatternSummary` — a derived, non-destructive, fully reconstructible artifact containing detected pattern states across approved dimensions.

M5.1 does not:
- present data to the user
- modify session history
- modify Body Context
- modify M4 intake, engine, or safety logic
- make adaptive decisions itself

M5.1 produces signals. M5.2 and M5.3 are permitted to consume those signals within their own contracted limits.

---

## 2. Input Source and Eligibility

### 2.1 Sole permitted read source
M5.1 reads exclusively from `getEligibleHariHistory()`.

It must never read from:
- `loadHistory()` directly
- `pending` sessions
- `invalidated` sessions
- Body Context (Truth Class A)
- Active Session State (Truth Class B)

### 2.2 Minimum session requirement
Pattern detection requires at least `CANDIDATE_MIN_RAW_COUNT` (2) eligible sessions.

If fewer than 2 eligible sessions exist, M5.1 must return a valid `PatternSummary` with all dimension entries set to `null`. All `null` dimensions are semantically treated as `insufficient`. No partial pattern signals may be emitted.

### 2.3 Eligible session shape requirement
An eligible session must have `hari_metadata` present and non-null. Sessions missing `hari_metadata` are excluded even if they pass the `isEligibleHariSession()` check.

This is a structural guard against malformed entries reaching the pattern reader.

---

## 3. Pattern Dimensions

A pattern dimension is a bounded, observable field drawn from eligible session data. Each dimension maps to one or more source fields and has a declared downstream use.

M5.1 detects patterns on exactly the following seven dimensions. No additional dimensions may be added without a contract amendment.

### 3.1 Dimension Independence Rule

Each pattern dimension must be evaluated independently.

No dimension may:
- influence the detection of another dimension
- combine scores with another dimension
- create composite or chained pattern logic

Cross-dimension reasoning is not permitted at M5.1.

| Uncertainty handling | `'not_sure'` observations have weight multiplied by `UNCERTAIN_VALUE_WEIGHT` (0.5) |

### Dimension 5 — `protocol_benefit_tendency`
| Property | Value |
|----------|-------|
| Source fields | `hari_metadata.intervention.mapped_protocol_id` + `result` |
| Derived value | Pair: `{ protocol_id: string; outcome_class: ProtocolOutcomeClass }` |
| Outcome mapping | `result === 'better'` → `'beneficial'`; `result === 'same'` → `'neutral'`; `result === 'worse'` or `'interrupted'` → `'adverse'` |
| Concordance rule | Only `'beneficial'` observations contribute to concordance for a given protocol. `'adverse'` observations for that protocol are treated as safety contradictions. |
| Dominant value | The `protocol_id` with the highest net weighted beneficial score. |
| Downstream use | M5.3 protocol hint reinforcement via existing `hariHintProtocolId` hook only |

---

### Dimension 6 — `pacing_tendency`
| Property | Value |
|----------|-------|
| Source fields | `hari_metadata.round_plan.breath_count` + length of `hari_metadata.reassessment_history` |
| Derived value | `RoundCountProfile`: `'minimal'` (≤10 breaths, 0–1 rounds) / `'short'` (20 breaths or 1–2 rounds) / `'standard'` (30 breaths or 2–3 rounds) / `'extended'` (3+ rounds) |
| Note | D1 debt — `round_plan` is currently singular. Until D1 migration is resolved, this dimension is computed from the initial `round_plan` only and must be treated as `candidate` maximum regardless of session count. This cap is lifted once the D1 migration is complete. |
| Downstream use | M5.3 round sizing advisory to M4.6 within existing safe bounds only |

---

### Dimension 7 — `outcome_trajectory`
| Property | Value |
|----------|-------|
| Source fields | `result` + pain delta (`hari_metadata.baseline_intensity - hari_metadata.post_session_intensity`) where both are present |
| Derived value | `OutcomeTrajectory`: `'improving'` / `'stable'` / `'variable'` / `'adverse'` |
| Trajectory rules | `'improving'`: majority of sessions with `result === 'better'` and positive pain delta. `'stable'`: majority `'same'` or small deltas. `'variable'`: mixed results with no dominant direction. `'adverse'`: recent `result === 'worse'` or `'interrupted'` in last 2 sessions. |
| Critical constraint | `outcome_trajectory` is observational only. It must never feed personalization decisions in M5.2 or M5.3. It is reserved for explainability display (M5.4) and internal pattern validity checks. |
| Diagnostic prohibition | Pain delta is a directional behavioral signal, not a clinical outcome measure. It may not be presented as evidence of treatment efficacy or medical improvement. |

## 4. Evidence Units

### 4.1 Definition
One evidence unit is a single eligible session contributing one observation to a single dimension.

Each eligible session contributes at most one observation per dimension. If a field is absent or null in a session's `hari_metadata`, that session contributes no observation to that dimension (it is skipped, not counted as a contradiction).

### 4.2 Concordant observation
An observation is concordant with a detected pattern if its value matches the dominant value for that dimension.

### 4.3 Contradicting observation
An observation is contradicting if its value differs from the dominant value for that dimension.

A contradicting observation on `protocol_benefit_tendency` where `outcome_class === 'adverse'` is classified as a **safety contradiction** and applies a higher penalty (see §6.3).

### 4.4 Uncertainty-bearing observation
An observation bearing an uncertain value (`'not_sure'`, `'mixed'`) is included in dimension analysis but its effective weight is multiplied by `UNCERTAIN_VALUE_WEIGHT` before contributing to concordance or contradiction scores.

---

## 5. Recency Weighting

Recency weighting is applied per-observation before any pattern state calculations. All thresholds in §6 operate on weighted scores.

### 5.1 Declared constants
```
RECENCY_DECAY_BASE    = 0.80   // positional decay factor
RECENCY_AGE_CAP_DAYS  = 90    // sessions older than this receive capped weight
RECENCY_AGE_CAP_WEIGHT = 0.25 // maximum weight for sessions beyond age cap
RECENCY_STALENESS_FLOOR = 0.15 // sessions with effective weight below this are excluded
```

### 5.2 Positional weight
Sessions are ordered by `timestamp`, most recent first (position 0).

```
positional_weight(position) = RECENCY_DECAY_BASE ^ position
```

Examples:
| Position | Positional weight |
|----------|------------------|
| 0 (most recent) | 1.00 |
| 1 | 0.80 |
| 2 | 0.64 |
| 3 | 0.51 |
| 4 | 0.41 |
| 5 | 0.33 |

### 5.3 Age cap
If a session's `timestamp` is more than `RECENCY_AGE_CAP_DAYS` (90) days before the computation date:

```
age_cap_weight = RECENCY_AGE_CAP_WEIGHT (0.25)
```

### 5.4 Effective weight
```
effective_weight = min(positional_weight, age_cap_weight)
```

### 5.5 Staleness exclusion
If `effective_weight < RECENCY_STALENESS_FLOOR` (0.15), the observation is excluded from pattern calculations entirely. It does not contribute to concordance, contradiction, or raw counts.

### 5.6 Flare sensitivity safety exception
For `flare_sensitivity_tendency`, observations with value `'high'` are never downweighted below `0.50` by recency. A recent `'high'` observation always retains significant weight. This prevents adaptive logic from quietly drifting toward a lower sensitivity assumption just because recent sessions were milder.

## 6. Contradiction Detection

### 6.1 Standard contradiction
A contradicting observation (§4.3) reduces the net weighted concordance score by its effective weight multiplied by `CONTRADICTION_PENALTY_MULTIPLIER`.

```
CONTRADICTION_PENALTY_MULTIPLIER = 2.0
```

Net weighted concordance formula:
```
net_weighted_concordance =
  sum(effective_weight for concordant observations)
  - sum(effective_weight × CONTRADICTION_PENALTY_MULTIPLIER for contradicting observations)
```

### 6.2 Recent contradiction flag
If any contradicting observation exists at position 0 or position 1 (the two most recent eligible sessions), the `DetectedPattern` field `has_recent_contradiction` is set to `true`.

A pattern with `has_recent_contradiction = true` may not exceed `candidate` state, regardless of total weighted score or raw count.

### 6.3 Safety contradiction
A safety contradiction is any of:
- `outcome_class === 'adverse'` on `protocol_benefit_tendency`
- `result === 'worse'` on the source `HistoryEntry`
- `result === 'interrupted'` on the source `HistoryEntry`
- Any `reassessment_history` entry where `response === 'worse'`

Safety contradictions apply a higher penalty:
```
SAFETY_CONTRADICTION_MULTIPLIER = 3.0
```

Additionally, a safety contradiction at position 0 (the most recent session) immediately sets the affected dimension's state to `suspended`, regardless of other scores.

### 6.4 Contradiction on `flare_sensitivity_tendency`
For flare sensitivity, contradictions toward higher sensitivity (e.g., pattern is `'low'` but recent session reported `'high'`) are treated as safety contradictions, not standard contradictions. The system must always err toward greater caution.

## 7. Pattern States

Each dimension produces exactly one of the following states. All state gates operate on the weighted scores and raw counts computed under §5 and §6.

### Declared thresholds
```
CANDIDATE_MIN_RAW_COUNT           = 2
ACTIVE_MIN_RAW_COUNT              = 3
ACTIVE_MIN_NET_WEIGHTED_SCORE     = 1.8
HIGH_CONFIDENCE_MIN_RAW_COUNT     = 5
HIGH_CONFIDENCE_MIN_NET_WEIGHTED_SCORE = 3.0
HIGH_CONFIDENCE_CLEAN_WINDOW      = 3   // no contradictions in last N concordant sessions
```

---

### `insufficient`
**Condition:** raw concordant count < `CANDIDATE_MIN_RAW_COUNT` (2).

No pattern is recognized. The dimension value is `null`.
No downstream consumer may use this dimension.

---

### `candidate`
**Conditions:**
- raw concordant count ≥ `CANDIDATE_MIN_RAW_COUNT` (2)
- no `suspended` trigger (§7 `suspended` rules)

**Permitted use:**
- May inform phrasing and explanation style only
- Must not influence intake defaults, protocol selection, or round sizing
- Authority: M5.0 §2.3

---

### `active`
**Conditions:**
- raw concordant count ≥ `ACTIVE_MIN_RAW_COUNT` (3)
- `net_weighted_concordance` ≥ `ACTIVE_MIN_NET_WEIGHTED_SCORE` (1.8)
- `has_recent_contradiction` is `false`
- no `suspended` trigger

**Permitted use:**
- May inform intake soft defaults (M5.2)
- May inform protocol hint (M5.3)
- May inform round sizing advisory (M5.3, pacing dimension only)
- Authority: M5.0 §2.1 Active Pattern definition

---

### `high_confidence`
**Conditions:**
- raw concordant count ≥ `HIGH_CONFIDENCE_MIN_RAW_COUNT` (5)
- `net_weighted_concordance` ≥ `HIGH_CONFIDENCE_MIN_NET_WEIGHTED_SCORE` (3.0)
- `has_recent_contradiction` is `false`
- no contradictions in the `HIGH_CONFIDENCE_CLEAN_WINDOW` (3) most recent concordant sessions
- no `suspended` trigger

**Permitted use:**
- All `active` uses, with stronger weighting permitted where downstream contracts allow
- Authority: M5.0 §2.1 High-Confidence Pattern definition

---

### `suspended`
**Trigger conditions (any one is sufficient):**
- safety contradiction at position 0 (most recent session)
- `has_recent_contradiction` is `true` AND previous state was `active` or `high_confidence`

**Effect:**
- Pattern is recognized as previously existing but currently paused
- No downstream consumer may act on a `suspended` pattern
- `dominant_value` is preserved in storage for continuity tracking
- State may recover to `active` or `candidate` if subsequent sessions reinforce concordance and the contradiction recedes in positional weight

**Authority:** M5.0 §2.2 ("Contradictory evidence weakens or suspends pattern use"), §9.8 (Pattern Inertia)

### 7.1 Pattern Usage Guard

No consumer may use a pattern unless:
- `pattern.state` is `'active'` or `'high_confidence'`
- pattern is not `'suspended'`
- no safety contradiction is active

Any attempt to use a pattern outside these conditions must be ignored.

## 8. Pattern Storage Schema

### 8.1 Truth class
`PatternSummary` is Truth Class D: Derived Pattern Intelligence.

It must never be written to:
- Body Context storage (Truth Class A)
- Active session state (Truth Class B)
- Session history records (Truth Class C)

It is stored in its own isolated storage key: `hari_pattern_summary`.

### 8.2 Non-destructive guarantee
`PatternSummary` is fully reconstructible from `getEligibleHariHistory()` at any time.

It contains no information not derivable from eligible session history. Deleting the stored `PatternSummary` causes no data loss. Clearing history automatically invalidates the summary.

### 8.3 Cache invalidation rules
The cached `PatternSummary` must be discarded and recomputed when:
- A new session is validated (validation_status transitions to `validated`)
- A session is deleted from history
- A session's validation status changes
- `source_session_ids` in the cached summary no longer matches the current `getEligibleHariHistory()` session IDs

If the cache is stale, the consumer must either trigger recomputation or treat all dimensions as `insufficient` until the cache is refreshed.

### 8.4 PatternSummary schema
```typescript
// Truth Class D — Derived Pattern Intelligence
// Authority: M5.1 §8
// Storage key: 'hari_pattern_summary'

interface PatternSummary {
  schema_version: 1
  computed_at: string            // ISO 8601
  source_session_count: number   // count of sessions used in computation
  source_session_ids: string[]   // for cache validation
  patterns: {
    symptom_focus_tendency:    DetectedPattern<SymptomFocus>        | null
    flare_sensitivity_tendency: DetectedPattern<FlareSensitivity>   | null
    session_intent_tendency:   DetectedPattern<SessionIntent>       | null
    session_length_tendency:   DetectedPattern<SessionLengthPreference> | null
    protocol_benefit_tendency: DetectedPattern<string>              | null  // dominant protocol_id
    pacing_tendency:           DetectedPattern<RoundCountProfile>   | null
    outcome_trajectory:        DetectedPattern<OutcomeTrajectory>   | null
  }
}

interface DetectedPattern<T> {
  dimension: PatternDimension
  state: PatternState
  dominant_value: T
  raw_concordant_count: number
  net_weighted_concordance_score: number
  has_recent_contradiction: boolean   // true if contradiction at position 0 or 1
  plain_summary: string               // max 80 chars, non-diagnostic
}

type PatternState =
  | 'insufficient'
  | 'candidate'
  | 'active'
  | 'high_confidence'
  | 'suspended'

type PatternDimension =
  | 'symptom_focus_tendency'
  | 'flare_sensitivity_tendency'
  | 'session_intent_tendency'
  | 'session_length_tendency'
  | 'protocol_benefit_tendency'
  | 'pacing_tendency'
  | 'outcome_trajectory'

type OutcomeTrajectory =
  | 'improving'
  | 'stable'
  | 'variable'
  | 'adverse'

type ProtocolOutcomeClass =
  | 'beneficial'
  | 'neutral'
  | 'adverse'
```

### 8.5 Versioning
`schema_version: 1` is the M5.1 baseline.

If the schema is modified in a future contract, version must be incremented and a migration path defined. Consumers must validate the schema version before reading. An unrecognized version must be treated as a stale cache (discard and recompute or return a valid `PatternSummary` with all dimension entries `null`, semantically treated as `insufficient`).

## 9. Explainability Requirements

### 9.1 All scoring inputs must be traceable
M5.1 must not produce a pattern state that cannot be traced back to:
- a specific set of source sessions (by session_id)
- declared weight constants (§5.1)
- declared threshold constants (§7)

No hidden scoring. No undeclared weighting adjustments.

### 9.2 `plain_summary` requirement
Every `DetectedPattern` must contain a `plain_summary` string of maximum 80 characters.

This string must be:
- written in plain language, not medical terminology
- non-diagnostic (describes behavior, not condition)
- accurate to the current state

Examples:
- `"Most sessions address neck / upper region"` (symptom_focus_tendency, active)
- `"Flare sensitivity frequently reported as high"` (flare_sensitivity_tendency, active)
- `"Session length preference varies — no clear pattern"` (session_length_tendency, insufficient)
- `"Recent session contradicts prior trend — pattern paused"` (any, suspended)

### 9.3 `outcome_trajectory` non-diagnostic language
`plain_summary` for `outcome_trajectory` must describe the directional observation only.

Permitted: `"Most sessions reported as improved"`, `"Mixed outcomes across recent sessions"`

Prohibited: `"Treatment is effective"`, `"Condition is improving"`, `"Protocol resolved symptoms"`

### 9.4 Downstream consumers must expose state
Any M5.2 or M5.3 behavior derived from a `DetectedPattern` must be queryable — meaning the state, dimension, and dominant_value driving the behavior must be accessible to a future explainability surface.

The system must be structurally incapable of applying adaptation that cannot be explained.

---

## 10. Non-Diagnostic Constraints

### 10.1 Pattern dimensions describe behavior, not condition
Every detectable dimension reflects a user's interaction choice (what they selected during intake) or a directional outcome signal (better/same/worse), not a clinical finding.

`symptom_focus` is what the user selected as their session focus — not a diagnosis.
`flare_sensitivity` is how the user reported feeling — not a clinical sensitivity measurement.
`outcome_trajectory` is the direction of self-reported experience — not a treatment outcome.

### 10.2 Prohibited inferences
M5.1 patterns must never be used to infer or imply:
- underlying injury or structural cause
- diagnosis or diagnostic probability
- prognosis or recovery trajectory
- medical clearance or contraindication
- severity classification of the user's condition

### 10.3 `outcome_trajectory` isolation
`outcome_trajectory` exists to support future display (M5.4) and to inform pattern validity checks (whether worsening outcomes should suspend other patterns).

It must not feed into M5.2 intake defaults or M5.3 protocol/pacing hints directly.

## 11. Public API Contract

M5.1 exposes the following function signatures. These define what M5.2, M5.3, and M5.4 may call. No other access to pattern internals is permitted.

```typescript
/**
 * Compute a fresh PatternSummary from eligible history.
 * Always produces a valid PatternSummary — returns all dimension entries
 * as null if history is empty or below minimum session threshold.
 * All null dimensions are semantically treated as 'insufficient'.
 * Authority: M5.1
 */
function computePatternSummary(): PatternSummary

/**
 * Load the cached PatternSummary from storage.
 * Returns null if no cache exists or if cache is stale.
 * Consumers must call computePatternSummary() if null is returned.
 * Authority: M5.1 §8.3
 */
function loadCachedPatternSummary(): PatternSummary | null

/**
 * Save a computed PatternSummary to the isolated storage key.
 * Must only be called with a freshly computed summary.
 * Authority: M5.1 §8
 */
function savePatternSummary(summary: PatternSummary): void

/**
 * Invalidate (delete) the cached PatternSummary.
 * Must be called whenever eligible history changes.
 * Authority: M5.1 §8.3
 */
function invalidatePatternSummaryCache(): void

/**
 * Convenience accessor — returns a PatternSummary from cache if valid,
 * or computes and caches a fresh one.
 * This is the primary consumer-facing entry point.
 * Authority: M5.1
 */
function getOrComputePatternSummary(): PatternSummary
```

### 11.1 Consumer rules
- M5.2 may only read `PatternSummary.patterns` from the output of `getOrComputePatternSummary()`
- M5.2 may only act on dimensions with state `active` or `high_confidence`
- M5.3 may only act on `protocol_benefit_tendency` and `pacing_tendency` with state `active` or `high_confidence`
- M5.4 may read any dimension for display, including `candidate` and `outcome_trajectory`
- No consumer may modify a `PatternSummary` — it is read-only output

## 12. Graceful Degradation

M5.1 must fail safely. Under every failure condition, behavior must revert to M4 baseline.

| Condition | Required behavior |
|-----------|------------------|
| `getEligibleHariHistory()` returns empty | Return a valid `PatternSummary` with all dimension entries `null`; consumers treat all `null` dimensions as `insufficient` |
| Fewer than 2 eligible sessions | Same as above |
| `hari_metadata` absent on an otherwise eligible session | Skip that session silently |
| All sessions are stale (all weights below floor) | Return a valid `PatternSummary` with all dimension entries `null`; consumers treat all `null` dimensions as `insufficient` |
| Storage read/write failure | Compute from history on each call; log failure; do not crash |
| Schema version mismatch on cache read | Discard cache, recompute |
| Any computation error | Return a valid `PatternSummary` with all dimension entries `null`; consumers treat all `null` dimensions as `insufficient`; do not propagate error to caller |

## 13. Carry-Forward Resolution

### D1 — `round_plan` singular
Until D1 debt is resolved (migration of `round_plan: RoundPlan` → `round_plan: RoundPlan[]`), the `pacing_tendency` dimension is capped at `candidate` state maximum. This cap is declared explicitly in the `DetectedPattern` for `pacing_tendency` via a `note` field added at D1-resolution time. No other dimension is affected by D1.

### D3 — Exit gating
M5.1 has no dependency on D3. D3 is a GuidedSessionScreen UX issue. It must be resolved before M5.2 is implemented (per M4.9 prerequisites) but does not affect this contract.

## 14. What This Contract Does Not Authorize

This contract does not authorize:

- implementing `computePatternSummary()` or any logic described here
- writing `PatternSummary` to storage
- modifying `sessionHistory.ts`, `hari.ts`, or any M4 file
- presenting pattern data to the user
- using pattern data to influence session intake, protocol selection, or round sizing
- any M5.2 or M5.3 behavior

Those require their own scoped contracts, each compliant with M5.0 and this contract.
