M5.0 — Adaptive Intelligence Identity Contract

Status: Boundary contract only
Purpose: Define the identity, limits, and operating rules of HARI’s adaptive intelligence layer without implementing learning behavior.

0. Foundational Authority

This contract governs all future M5 layers.

Anything that conflicts with this contract is invalid, even if technically feasible.

HARI’s intelligence layer must remain:

safe
bounded
explainable
non-diagnostic
user-controlled
conservative under uncertainty

This contract does not authorize:

pattern learning implementation
M4 modification
hidden adaptation
raw-history use
validation bypass
1. What Qualifies as a Pattern

A pattern is a bounded, evidence-supported tendency observed across validated sessions and used only within approved adaptive constraints.

A pattern must be:

repeated rather than one-off
derived from validated outputs rather than raw history
behavioral or interactional, not diagnostic
relevant to regulation delivery, not identity speculation
reversible if future evidence weakens it

A pattern may describe only things such as:

repeated response preference
repeated cue tolerance
repeated pacing preference
repeated benefit from a previously approved intervention order
repeated need for more conservative progression

A pattern is not:

a single event
a severe moment treated as identity truth
unfiltered self-report alone
a diagnosis
a structural claim
a hidden trait assignment
a conclusion drawn from raw history text
2. Evidence Thresholds

HARI may only recognize pattern status through derived evidence summaries produced after M4 validation.

2.1 Pattern States

Candidate Pattern
May be marked only when there are at least 2 concordant validation-passed observations from separate completed cycles.

Active Pattern
May be used for bounded personalization only when there are at least 3 concordant validation-passed observations from separate completed cycles and no unresolved contradictory evidence of similar strength.

High-Confidence Pattern
May be weighted more strongly only when there are at least 5 concordant validation-passed observations across time with continued stability and no recent contradictory evidence.

2.2 Evidence Rules
Validation-passed evidence counts; rejected or filtered evidence does not.
Raw history does not count as direct adaptive evidence.
Contradictory evidence weakens or suspends pattern use.
Sparse evidence must be treated as uncertainty, not support.
Absence of evidence is never evidence of preference, tolerance, or safety.
Safety-relevant contradictions immediately reduce personalization confidence.
Recency Rule

More recent validated sessions must carry more weight than older sessions.

If newer evidence contradicts older patterns:

pattern confidence must be reduced or suspended

Patterns must decay over time if not reinforced.

2.3 Pattern Detection vs Pattern Usage

Pattern detection and pattern usage must be treated as separate steps.

Detection means:

identifying that a pattern may exist
occurring once evidence thresholds are met

Usage means:

allowing that pattern to influence behavior

A pattern may be detected before it is used.

Usage is only permitted when:

pattern meets Active Pattern threshold or higher
no safety contradiction exists
current session state supports its application

Low-confidence patterns must not influence decisions directly.
They may only influence explanation or phrasing.

3. What HARI May Personalize

HARI may personalize delivery, not diagnosis.

Allowed personalization is limited to already approved, M4-compatible behavior such as:

wording simplicity or specificity
cue density
pacing within approved bounds
ordering among already approved interventions
repetition level
explanation brevity
check-in style
conservative preference weighting
reminder style
preference for gentler vs more direct presentation, when safe

HARI may also personalize toward greater caution when evidence suggests lower tolerance or increased sensitivity.

HARI may not create new interventions, new medical claims, or hidden treatment logic.

3.1 Application Boundaries

Personalization may only influence:

M4.2 intake defaults (pre-population only)
M4.5 intervention selection weighting (within existing options)
M4.6 round sizing adjustments (within safe bounds)
phrasing and explanation style

Personalization must not modify:

safety gate logic
validation rules
eligibility filtering
intervention class definitions
persistence structure
4. What HARI Must Never Infer

HARI must never infer, state, or imply:

diagnosis
injury identity
structural displacement
nerve damage certainty
psychological diagnosis
root-cause certainty
medical clearance
recovery timeline certainty
pain severity certainty from behavior alone
intent, motivation, or character from symptom behavior
safety from silence
consent from prior compliance
hidden readiness beyond observed evidence

HARI must also never infer identity-level conclusions from adaptive behavior.

It may describe observed interaction tendencies, not underlying medical or personal truth.

5. How Recommendations Evolve Safely

Recommendations may evolve only through bounded reweighting inside approved M4 space.

Safe evolution means:

no new intervention class is introduced
no safety rule is weakened
no validation filter is bypassed
no personalization becomes irreversible
no recommendation is upgraded from weak evidence alone
changes remain explainable in plain language

Recommendation evolution must occur through:

ordering adjustments
pacing adjustments
cueing adjustments
conservative weighting of previously validated options
fallback to neutral defaults when evidence is weak or conflicted

Adaptation must be incremental, not abrupt.

When uncertain, HARI must adapt by becoming less assumptive, not more complex.

6. How Uncertainty Limits Adaptation

Uncertainty is a hard limiter.

When evidence is sparse, mixed, stale, filtered, or contradictory, HARI must:

reduce confidence
narrow personalization
revert toward neutral defaults
preserve optionality
state uncertainty honestly
avoid specificity it cannot support

Uncertainty must never be converted into hidden inference.

Low-confidence patterns may inform phrasing, but they may not drive strong recommendation shifts.

When confidence falls below Active Pattern threshold, adaptive use must pause.

7. How User Override Is Preserved

User control remains primary unless safety requires restriction.

The user must always be able to:

decline a recommendation
stop or pause
choose a different approved option
request neutral or non-personalized mode
reset personalization state
ask why a recommendation is being offered
override preference assumptions

Explicit user choice outranks inferred preference.

HARI may remember bounded adaptive tendencies in future design, but it may not lock the user into them.

No adaptive pathway may become hidden, mandatory, or hard to escape.

8. How Safety Overrides Personalization

Safety always outranks personalization.

If a safety trigger, escalation condition, adverse response, or validation failure occurs:

personalization must immediately yield to the safest M4-compliant pathway
adaptive weighting must be suspended where relevant
no preference memory may justify continuation
no personalization may suppress caution, stopping, or reassessment

Personalization may only make the system more cautious, never less safe.

It must never:

lower stop sensitivity
reduce validation strictness
weaken reassessment
reinterpret adverse response as acceptable
override safety messaging
push continuation because a pattern “usually works”
9. Anti-Drift Rules for All Future M5 Layers

All future M5 work must obey the following:

9.1 Boundary Integrity
M5 may personalize only within explicitly allowed fields.
M5 may not modify M4 logic, gates, selectors, validators, or safety behavior.
9.2 No Hidden Adaptation
no silent weighting changes
no invisible pattern scoring
no unexplained recommendation shifts
no background adaptation outside declared mechanisms
9.3 No Raw-History Dependence
Raw history must not be used directly for adaptive decisions.
Only derived, validation-safe summaries may inform future adaptive logic.
9.4 No Diagnostic Drift
Adaptive intelligence must remain non-diagnostic at every layer.
Repeated behavior must not be converted into medical conclusions.
9.5 Explainability Requirement

Every adaptive effect must be traceable to:

allowed evidence type
confidence level
approved personalization field
reversible decision logic
9.6 Conservative Failure Mode

When a future M5 layer is uncertain, broken, conflicted, or incomplete, it must fail into:

neutral defaults
preserved user control
intact M4 safety behavior
9.7 No Scope Expansion by Implementation

Future implementation details may not quietly expand authority beyond this contract.

If a capability is not explicitly permitted here, it is out of scope.

9.8 Pattern Inertia

The system must not continue using an outdated pattern despite new contradictory evidence.

Prevention:

enforce recency weighting
allow pattern suspension
require ongoing reinforcement for continued use
10. Binding Identity Statement

HARI’s intelligence layer is a bounded adaptive delivery system, not a diagnostic engine.

It may learn only safe, explainable, validation-grounded delivery tendencies inside approved limits.

It must remain:

transparent in effect
conservative under uncertainty
reversible in personalization
subordinate to safety
subordinate to user control
structurally unable to drift into hidden inference
11. Non-Authorization Clause

This contract defines identity and limits only.

It does not authorize:

pattern learning implementation
memory schema changes
scoring logic
adaptation execution
raw-history processing
hidden personalization
M4 behavioral modification

Those require future scoped M5 layers and must remain compliant with this contract.
