# M5.4 — Session Insights Surface Contract

## Status
Definition layer for optional insight and explainability surface.

This contract defines how HARI may surface pattern-derived insights to the user.

M5.4 is:
- optional for MVP
- non-intrusive
- non-diagnostic
- explainability-focused only

No adaptive behavior may originate from this layer.

---

## 1. Purpose

M5.4 exists to:

- make HARI’s adaptive behavior transparent
- allow users to understand why suggestions appear
- reinforce trust without increasing complexity

M5.4 does NOT:
- change system behavior
- drive decisions
- introduce new intelligence
- interpret user condition

---

## 2. Core Principle

> Insights explain behavior — they do not define it.

Insights must:
- describe observed tendencies
- remain non-diagnostic
- remain reversible
- remain optional

---

## 3. Input Source

M5.4 reads only from:

```ts
getOrComputePatternSummary()

No raw history access is permitted.

4. Eligible Pattern States

M5.4 may display:

candidate
active
high_confidence
suspended

However:

candidate → must be labeled as low-confidence
suspended → must be clearly indicated as paused
insufficient → must not be shown
5. Allowed Insight Types

M5.4 may present:

5.1 Pattern summaries

Examples:

"You often choose neck or upper region focus."
"Recent sessions suggest higher sensitivity."
"Shorter sessions tend to be better tolerated."
5.2 Reinforcement explanation

Examples:

"This is suggested based on your recent validated sessions."
"A similar approach has helped recently."
5.3 Safety-based explanation

Examples:

"We're staying more cautious based on recent responses."
"A previous pattern is paused due to recent changes."
6. Prohibited Insight Types

M5.4 must NOT present:

diagnosis
injury claims
structural explanations
causal claims
prognosis
recovery timelines
severity labeling

Forbidden examples:

"You have a cervical issue"
"This is caused by nerve compression"
"Your condition is improving"
7. Tone Requirements

Insights must be:

calm
neutral
observational
non-authoritative

Avoid:

certainty language
clinical tone
persuasive framing
8. Placement Rules

Insights must be:

secondary UI elements
never blocking
never required
dismissible or ignorable

Possible locations:

below intake
after session completion
optional “Why this?” link
9. User Control

User must be able to:

ignore insights
proceed without interaction
not depend on insights to use system

Insights must never:

force attention
interrupt flow
gate progression
10. Explainability Rule

Every insight must be traceable to:

pattern dimension
pattern state
allowed input field
validated session evidence

No insight may appear without a clear source.

11. Failure Modes to Prevent
A — Diagnostic drift

Insights imply condition or injury

B — Overconfidence

Weak patterns presented as strong truth

C — Hidden logic

User cannot understand why suggestion appears

D — Cognitive overload

Too many insights shown

E — Behavioral influence

Insights pressure user decisions

12. Graceful Degradation

If:

PatternSummary is empty
no active patterns exist
computation fails

Then:

show no insights
system behaves normally
13. Integration Rules

M5.4 must:

never modify M4 or M5 logic
never alter decision-making
only read PatternSummary
remain purely presentational
14. Output Shape

Example structure:

Insight {
  dimension: PatternDimension
  state: PatternState
  summary: string
  confidence: 'low' | 'moderate' | 'high'
}
15. Final Instruction to Claude Code

Implement M5.4 as an optional explainability layer.

Do NOT:

influence behavior
modify system logic
introduce new inference

Ensure:

insights are minimal
insights are accurate
insights are optional
insights remain non-diagnostic
End of Contract
