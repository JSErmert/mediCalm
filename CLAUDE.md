Treat MediCalm’s docs as the source of truth for:
- product identity
- source-boundary rules
- safety behavior
- runtime logic
- UI/UX behavior
- copy and tone
- claims limitations

Build priorities:
1. Home + pain input
2. Deterministic session selection engine
3. Guided session UI
4. Completion + history save
5. Safety interruption flow
6. Reduced motion + accessibility support
7. Bounded personalization

Core rules:
- MediCalm is a bounded symptom-support system
- It is not a diagnosis engine
- It is not a generic meditation or wellness app
- Safety outranks aesthetics, continuity, personalization, and follow-up
- When uncertain, choose the lower-risk implementation
- Keep the app mobile-first, low-overload, calm, medically grounded, and luminous but restrained
- Preserve countdown-first breathing, 3-step emergence, restrained crossfades, completion-to-history flow, and concise calm medical copy
- Avoid hype, mysticism, overpromising, false reassurance, and unsupported medical claims

If there is any conflict:
1. follow the MediCalm document hierarchy map
2. prefer safety over elegance
3. prefer clarity over novelty
4. prefer source-boundary discipline over confident invention
